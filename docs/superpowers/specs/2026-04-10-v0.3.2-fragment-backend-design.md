# v0.3.2 碎片观点库后端设计说明

## 结论摘要

- 本次后端采用现有三层架构：`Route Handler -> FragmentService -> FragmentRepository -> Prisma`。
- 新增底层模型 `Fragment`，对外仍使用产品语义路径 `/api/viewpoints`。
- `GET /api/viewpoints` 使用组织隔离 + 关键词搜索 + 复合 cursor 分页。
- `POST /api/viewpoints` 采用事务内逐条创建，保证返回的 `items` 精确对应本次成功创建的数据。
- `DELETE /api/viewpoints/:id` 采用软删除；创建者可删本人观点，`BRANCH_MANAGER` 与 `SUPER_ADMIN` 可删组织内任意观点。
- Repository 不承担权限判断，所有角色与组织校验集中在 Service 层。
- 本轮按完整后端交付执行：包含 schema、迁移、类型、repository、service、route、测试与验证。

## 方案取舍

本次讨论过 3 种后端实现方式：

1. 保守稳态方案：三层架构 + 事务内逐条创建 + 精确返回创建结果。
2. 性能优先方案：`createMany + 回查`，批量插入更快，但返回结果与本次请求的对应关系较弱。
3. 最小实现方案：仅打通接口，不完整覆盖分层与测试。

最终选择方案 1，原因有 3 点：

- 现有仓库已经稳定采用 Route / Service / Repository 分层，新功能继续沿用成本最低。
- 需求明确要求 `POST` 返回可直接用于前端刷新的创建结果，逐条创建最容易保证语义精确。
- 单次最多 50 条观点，逐条创建的性能成本在当前规模下可接受，换来更稳定的行为与更直接的测试断言。

## 模块边界

本次后端实现拆分为以下模块：

- `prisma/schema.prisma`
  - 新增 `Fragment` 模型。
  - 在 `Organization` 上增加 `fragments` 关联。
  - 在 `User` 上增加 `createdFragments` 关联，并使用命名 relation `FragmentCreator`。
- `src/types/fragment.ts`
  - 定义 `FragmentDTO`、`CreateFragmentsInput`、`ListFragmentsParams`、`CreateFragmentsResult`。
- `src/types/api.ts`
  - 追加 `CursorPaginatedData<T>`。
- `src/server/repositories/fragment.repository.ts`
  - 仅负责数据库读写与 cursor 查询拼装。
- `src/server/services/fragment.service.ts`
  - 负责输入清洗、业务校验、权限判断、DTO 映射。
- `src/app/api/viewpoints/route.ts`
  - 提供 `GET /api/viewpoints` 与 `POST /api/viewpoints`。
- `src/app/api/viewpoints/[id]/route.ts`
  - 提供 `DELETE /api/viewpoints/:id`。

## 数据模型

新增 Prisma 模型：

```prisma
model Fragment {
  id              String       @id @default(cuid())
  content         String       @db.Text
  organizationId  String
  organization    Organization @relation(fields: [organizationId], references: [id])
  createdByUserId String
  createdByUser   User         @relation("FragmentCreator", fields: [createdByUserId], references: [id])
  createdAt       DateTime     @default(now())
  updatedAt       DateTime     @updatedAt
  deletedAt       DateTime?

  @@index([organizationId])
  @@index([createdByUserId])
  @@map("fragments")
}
```

关联补充：

- `Organization.fragments Fragment[]`
- `User.createdFragments Fragment[] @relation("FragmentCreator")`

模型语义约束：

- `content` 使用 `@db.Text`，长度限制保持在业务层实现，避免数据库层约束影响后续扩展。
- `deletedAt` 用于软删除，所有列表与详情查询默认只看未删除数据。
- 组织隔离继续依赖 `organizationId` 显式过滤，不引入隐式全局作用域。

## API 与数据流

### `GET /api/viewpoints`

用途：

- 获取当前用户所在组织的观点列表。
- 支持按 `q` 做全文包含搜索。
- 支持基于 `createdAt + id` 的稳定 cursor 分页。

请求参数：

```typescript
z.object({
  q: z.string().max(200).optional(),
  cursor: z.string().optional(),
  limit: z.coerce.number().int().min(1).max(100).default(20),
});
```

数据流：

1. Route Handler 调用 `auth()`。
2. 使用 `requireRole(session, UserRole.EMPLOYEE, UserRole.BRANCH_MANAGER, UserRole.SUPER_ADMIN)`。
3. 解析 query 并交给 `fragmentService.listFragments(session.user, query)`。
4. Service 将 `caller.organizationId` 传入 Repository。
5. Repository 按 `createdAt desc, id desc` 查询，附带 `createdByUser` 精简信息。
6. Service 将结果映射为 `CursorPaginatedData<FragmentDTO>`。

分页规则：

- cursor 编码格式：`base64url(JSON.stringify({ c, i }))`
- 解码后字段：
  - `c`: 最后一条记录的 `createdAt.toISOString()`
  - `i`: 最后一条记录的 `id`
- 游标条件：

```typescript
OR: [
  { createdAt: { lt: cursorCreatedAt } },
  {
    AND: [{ createdAt: cursorCreatedAt }, { id: { lt: cursorId } }],
  },
];
```

### `POST /api/viewpoints`

用途：

- 批量录入观点。
- 返回本次真实创建成功的条目列表。

请求体：

```typescript
z.object({
  contents: z.array(z.string().min(1).max(500)).min(1).max(50),
});
```

数据流：

1. Route Handler 完成登录校验与 Zod 校验。
2. Service 对 `contents` 做 `trim()`。
3. 过滤空行。
4. 再次做业务层长度校验；若任一有效条目超过 500 字，抛出 `AppError("VALIDATION_ERROR", ...)`。
5. 若过滤后无有效内容，抛出 `AppError("VALIDATION_ERROR", ...)`。
6. Repository 在事务内逐条执行 `create`，每条都返回 `createdByUser`。
7. Service 映射为 `CreateFragmentsResult` 并返回。

本接口不采用 `createMany`，原因：

- 需要精确返回本次创建的 `items`。
- 允许重复内容，回查无法稳定区分“本次创建”与“历史同内容记录”。
- 上限 50 条时，事务内逐条创建可接受。

### `DELETE /api/viewpoints/:id`

用途：

- 软删除单条观点。

数据流：

1. Route Handler 校验登录状态并解析 `id`。
2. Service 先查找未删除的观点记录。
3. 若不存在，返回 `404 NOT_FOUND`。
4. Service 做角色与组织校验。
5. Repository 更新 `deletedAt`。
6. 返回 `{ id }`。

权限规则：

- `EMPLOYEE`：仅可删除 `createdByUserId === caller.id` 的记录。
- `BRANCH_MANAGER`：可删除当前组织内任意记录。
- `SUPER_ADMIN`：本版本仍按当前 session 的 `organizationId` 作用域处理，不开放跨组织删除。

## Repository 设计

`FragmentRepository` 提供以下公共方法：

- `findMany(params)`
  - 入参：`organizationId`、`q?`、`cursor?`、`limit`
  - 返回：`CursorPaginatedData<FragmentRecord>`
- `create(data, db?)`
  - 单条创建，支持事务客户端透传。
- `createManyPrecisely(items)`
  - 内部开启事务并逐条调用 `create`，返回完整已创建记录。
- `findById(id)`
  - 查询未删除观点，包含 `createdByUser`。
- `softDelete({ id, organizationId })`
  - 仅在组织命中时写入 `deletedAt`。

Repository 约束：

- 不判断用户角色。
- 所有查询默认增加 `deletedAt: null`。
- `include` 统一为 `createdByUser: { select: { id: true, name: true } }`，避免多处漂移。

## Service 设计

`FragmentService` 负责 3 个主方法：

- `listFragments(caller, params)`
  - 委托 Repository 查询，并映射 DTO。
- `createFragments(caller, input)`
  - 做 trim、空行过滤、长度校验、事务创建、结果映射。
- `deleteFragment(caller, fragmentId)`
  - 查询、权限判断、软删除。

Service 关键规则：

- 所有方法都以 `caller.organizationId` 为组织边界。
- 只要调用方已通过 Route 层 `requireRole`，Service 不再重复做登录判断。
- 删除权限判定顺序：
  1. 记录不存在 -> `404`
  2. 记录不属于当前组织 -> `403`
  3. 若 caller 为 `EMPLOYEE` 且不是创建者 -> `403`
  4. 其他允许角色可继续删除

DTO 映射输出：

```typescript
interface FragmentDTO {
  id: string;
  content: string;
  organizationId: string;
  createdByUserId: string;
  createdByUser: {
    id: string;
    name: string;
  };
  createdAt: string;
}
```

`createdAt` 对外统一转成 ISO 字符串，保持与现有 API 风格一致。

## 错误处理

统一沿用 `AppError` 与 `handleApiError`：

- `UNAUTHORIZED` / 401：未登录
- `FORBIDDEN` / 403：无权访问或删除
- `NOT_FOUND` / 404：观点不存在或已删除
- `VALIDATION_ERROR` / 400：请求参数不合法，或清洗后的业务输入不合法

实现原则：

- 路由层负责结构校验。
- Service 负责业务语义校验。
- Repository 不抛业务语义错误，异常由上层统一处理。

## 测试与验证

本轮按完整后端交付执行，测试重点放在 Service 与 Route 两层。

### Service 测试

覆盖以下场景：

- 批量录入时 `trim + 空行过滤`
- 超过 500 字时报 `VALIDATION_ERROR`
- 允许重复内容录入
- 返回的 `created` 与 `items` 精确对应本次创建结果
- `EMPLOYEE` 删除自己的观点成功
- `EMPLOYEE` 删除他人观点返回 `403`
- `BRANCH_MANAGER` / `SUPER_ADMIN` 删除组织内观点成功
- 删除不存在或已删除观点返回 `404`

### Route 测试

覆盖以下场景：

- 未登录返回 `401`
- `GET` query 校验
- `POST` body 校验
- `DELETE` 参数透传与统一响应格式

## 实施边界

本设计只覆盖 v0.3.2 的后端主链路，不包含以下内容：

- `/viewpoints` 页面与仿写阶段观点选择区的前端实现
- 观点与 AI 仿写 prompt 的持久化绑定
- 标签、分类、导出、跨组织共享

该后端设计的目标，是为 v0.3.2 前端列表/选择能力提供稳定 API，同时为 v0.3.3 仿写生成接入提供可复用的观点数据入口。
