# v0.2.1.1 Backend Design

**Version:** v0.2.1.1  
**Date:** 2026-04-04  
**Scope:** Real crawler integration, `secUserId`, `VideoSnapshot`, storage skeleton, cross-account video list API

## Goal

Replace the mock crawler path introduced in `v0.2.1` with a real crawler REST integration while preserving the existing layered backend architecture and current permissions. This version also adds the persistence needed for future trend charts and cross-account video browsing.

## Scope

This design covers all backend tasks in `docs/product/versions/v0.2.1.1/tasks-backend.md`:

- `BE-001` schema changes
- `BE-002` real crawler service integration
- `BE-003` `SyncService` and account creation flow upgrade
- `BE-004` `VideoSnapshot` collection scheduler
- `BE-005` storage service skeleton
- `BE-006` `GET /api/videos`

Frontend homepage work is explicitly out of scope for this spec.

## Architecture

The existing layering stays unchanged:

- Route Handler: validates request, checks auth, returns standardized API payload
- Service: owns business logic, permission decisions, orchestration, crawler field mapping
- Repository: owns Prisma queries and organization-aware access helpers

New logic is added by extending the current Douyin account/video flows rather than introducing a parallel subsystem.

## Data Model

### DouyinAccount

Add nullable unique `secUserId`:

- supports new accounts created from real crawler data
- keeps old rows compatible
- allows old rows to backfill the identifier lazily during first sync

### DouyinVideo

Add:

- `tags String[] @default([])` as a reserved filtering field for future classification
- `snapshots VideoSnapshot[]` relation

### VideoSnapshot

Add append-only snapshot rows linked to `DouyinVideo.id`:

- one row per collection event
- stores play/like/comment/share metrics
- indexed by `videoId` and `(videoId, timestamp)`
- no soft delete, because snapshots are historical records

## Crawler Integration

`CrawlerService` is rewritten to call the real crawler server via `env.CRAWLER_API_URL`.

Methods:

- `getSecUserId(url)`
- `fetchUserProfile(secUserId)`
- `fetchVideoList(secUserId, cursor?)`
- `fetchCollectionVideos(secUserId)`
- `fetchOneVideo(awemeId)`

Rules:

- every raw crawler response is logged with full JSON using `[CrawlerService] <path> response: ...`
- crawler field mapping is normalized inside the service only
- requests keep retry behavior from `v0.2.1`
- timeout is 30 seconds
- `coverUrl` and `videoUrl` emit `待下载` logs for future storage integration

The collection endpoint is wrapped now for parity and future reuse, but remains unused by business flows in this version.

## Account Creation And Sync Flow

### Preview / Create

The preview flow becomes:

1. user submits `profileUrl`
2. backend calls `getSecUserId(profileUrl)`
3. backend calls `fetchUserProfile(secUserId)`
4. preview response returns normalized profile plus `secUserId`

The create flow stores `secUserId` directly with the account row. The frontend remains responsible for sending the previewed `secUserId` back to the create API.

### Account Info Sync

`SyncService.syncAccountInfo` changes to:

1. ensure `secUserId` exists
2. if missing, derive it from `profileUrl` and persist it
3. fetch latest profile by `secUserId`
4. update account profile fields and `lastSyncedAt`

### Video Sync

Video sync switches from fixed page-count crawling to incremental crawling:

- if the account has no local videos yet, fetch the first 10 items
- otherwise fetch batches of 4 new videos
- stop as soon as a known `videoId` is encountered
- hard stop after 10 batches to avoid runaway crawling

This keeps sync cost bounded while preserving correct “new videos first” behavior.

## Snapshot Collection

Add `runVideoSnapshotCollection()` to `SyncService`.

Flow:

1. list all active `DouyinVideo`
2. fetch fresh metrics by `video.videoId`
3. append a `VideoSnapshot`
4. update current counters on `DouyinVideo`
5. continue on per-video failure

The scheduler adds:

- `VIDEO_SNAPSHOT_CRON ?? "*/10 * * * *"`

This runs alongside the existing account info sync and video sync schedules.

## Storage Skeleton

Add `StorageService` with:

- `downloadAndStore(url, category): Promise<string>`

Current behavior:

- derive `storage/{category}/{YYYY-MM-DD}/{filename}`
- log `[StorageService] 待下载: ...`
- return the target path only

No actual file IO happens in this version.

## Cross-Account Video API

Add `GET /api/videos`.

Request params:

- `page`
- `limit`
- `accountId`
- `tag`
- `sort` in `publishedAt | likeCount`
- `order` in `asc | desc`

Service behavior:

- `EMPLOYEE`: only own accounts
- `BRANCH_MANAGER`: all accounts in own organization
- `SUPER_ADMIN`: all accounts
- if `accountId` is provided, access is revalidated against caller scope

Repository behavior:

- paginate videos
- include account nickname and avatar
- filter by `tags has <tag>` when provided
- support dynamic sorting

## Testing Strategy

Each backend task is implemented test-first and verified before moving on.

Required coverage:

- schema and env parsing updates
- crawler service method mapping and error handling
- account preview/create flow with `secUserId`
- sync service backfill + incremental sync behavior
- snapshot collection behavior
- `/api/videos` role filtering and parameter validation

Execution rule for this iteration:

- finish one backend task
- self-review that task against its checklist
- report the self-review result in chat
- then continue to the next task

## Risks And Decisions

### Real crawler response shapes may differ

Decision:

- implement first-pass mapping conservatively
- log full JSON for every endpoint
- keep mapping localized to `CrawlerService`

### Local Prisma migrate history may drift from the database

Decision:

- generate migration files in repo
- prefer non-destructive synchronization for the local environment if drift blocks `migrate dev`
- do not reset the database unless explicitly requested

### Production build requires crawler env

Decision:

- keep the current production guard in `env.ts`
- for local verification, provide a temporary `CRAWLER_API_URL` value when running `pnpm build`

## Acceptance

This design is complete when:

- mock crawler behavior is fully removed from business code
- `secUserId` is persisted and backfilled
- snapshots are collected on schedule
- `/api/videos` serves paginated role-filtered data
- existing `v0.2.1` manual sync and scheduler flows remain compatible
