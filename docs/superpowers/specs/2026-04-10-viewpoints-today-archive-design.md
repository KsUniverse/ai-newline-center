# 观点库今日/历史分层设计

> 日期：2026-04-10
> 适用范围：观点库页面、观点 API、AI 仿写阶段观点选择区

## 1. 背景

当前分支已经完成观点库基础能力，但还存在三个待处理问题：

1. 本地创建观点时报错：`P2021 The table fragments does not exist in the current database`
2. 观点库尚未区分“今日观点”和“历史观点”
3. AI 仿写阶段仍能看到全部观点，未限制为仅引用今日观点

本设计的目标是：

- 修复当前阻塞使用的数据库同步问题
- 将观点库主页面聚焦为“今日观点池”
- 提供独立的“历史归档”页面承载历史观点
- 将 AI 仿写阶段的观点引用范围限制为今日观点
- 在不偏离现有 UI/UX 品牌语言的前提下，提升观点浏览与选择交互的优雅度

## 2. 根因结论

`fragments` 报错不是业务代码错误，而是数据库结构未同步。

证据：

- `prisma/schema.prisma` 已定义 `Fragment`，并通过 `@@map("fragments")` 映射到 `fragments` 表
- `prisma/migrations/20260410130917_add_fragments/migration.sql` 已包含 `CREATE TABLE fragments`
- `src/server/repositories/fragment.repository.ts` 全程使用 `db.fragment`
- `src/app/api/viewpoints/route.ts` -> `fragmentService.createFragments()` -> `fragmentRepository.createManyPrecisely()` -> `db.fragment.create()` 的调用链完整一致

结论：

- 当前本地数据库未应用最新 migration
- 实现时应先同步数据库结构，再继续功能改造
- 不引入代码级“缺表兜底”逻辑

## 3. 设计原则

1. 今日/历史是时间视图，不是持久业务状态
2. 不为“今日/历史”新增冗余数据库字段
3. 主页面只展示今日观点，历史内容进入独立归档页
4. AI 仿写只暴露今日观点，避免历史噪音进入当前创作流
5. 沿用现有 `DashboardPageShell`、`MetaPillList`、surface section 语言
6. 参考 benchmark archive 的信息架构，但不机械复刻其卡片形态

## 4. 数据与接口设计

### 4.1 时间边界

“今日观点”按 `Asia/Shanghai` 自然日划分：

- 今日开始：当天 `00:00:00.000`
- 今日结束：次日 `00:00:00.000` 前

查询规则：

- `today`：`createdAt >= todayStart && createdAt < nextDayStart`
- `history`：`createdAt < todayStart`

### 4.2 数据模型

不新增 Prisma 字段，不修改 `Fragment` 表结构。

原因：

- 今日/历史仅由 `createdAt` 推导
- 避免多维护一份容易过期的状态字段
- 降低迁移和兼容成本

### 4.3 API 查询范围

扩展现有 `GET /api/viewpoints` 查询参数：

- `scope=today | history`
- 默认值：`today`

接口职责：

- 服务端负责按 scope 严格过滤
- 服务端继续返回现有 `FragmentDTO`
- 不在接口层增加展示型衍生字段，例如日期标题、日期标签

### 4.4 分页策略

延续现有 cursor 分页模型：

- 今日主列表：cursor + infinite scroll
- 历史归档页：cursor 查询，但前端按日期分组渲染

选择 cursor 的原因：

- 观点是短文本流，不是需要稳定页码跳转的实体档案表
- 与当前 `fragment.repository.ts` 实现兼容，改动更小

## 5. 页面与交互设计

### 5.1 主页面：今日观点池

路由保持为：

- `/viewpoints`

页面目标：

- 只承载“今天新增、今天可引用”的观点
- 快速浏览、录入、搜索、删除

页面结构：

1. `DashboardPageShell`
2. 顶部 actions
   - `历史归档`
   - `添加观点`
3. 今日概览 pills
   - 例如：`今日 18 条`
   - 例如：`仅展示今天新增`
   - 例如：`历史内容已归档`
4. 今日观点列表区

列表交互：

- 搜索仅在今日范围内生效
- 卡片继续保持短文本流式阅读
- 删除操作维持 hover 出现，避免抢主内容视觉焦点
- 空状态明确表达“今天暂无新增观点”

### 5.2 历史归档页

新增独立路由：

- `/viewpoints/archived`

交互参考：

- 复用 benchmark archive 的页面壳、返回入口与概览信息布局
- 不复用 benchmark 的档案卡片密度，而改用更适合短文本观点的按日归档清单

页面结构：

1. `DashboardPageShell`
2. 返回按钮：回到 `/viewpoints`
3. 归档概览 pills
4. 历史观点清单

历史观点清单展示方式：

- 按日期分组
- 日期倒序
- 每个日期组包含当天所有历史观点

日期组样式：

- 日期标题使用 eyebrow 风格小标题
- 日期组内部观点继续使用轻量 surface 卡片

### 5.3 添加观点交互

保留现有 `Dialog` 录入方式，不改为独立页面或大抽屉。

原因：

- 当前场景是短内容高频录入
- `Dialog` 已符合项目对短表单的交互约束
- 本轮重点应放在信息分层，而不是更换录入容器

### 5.4 仿写阶段：仅引用今日观点

AI 仿写阶段中，观点引用区块只加载：

- `scope=today`

交互改造目标：

- 从“功能型 checkbox 列表”升级为“创作参考池”
- 提升可读性与选中反馈
- 保持低干扰，不抢草稿编辑区注意力

区块调整：

- 标题改为“今日观点参考”
- 搜索仅搜索今日观点
- 列表项改为可点选小卡片，而非纯 checkbox 行
- 卡片信息包含：
  - 观点正文
  - 创建人
  - 创建时间
- 选中态使用边框、轻背景、强调条表达
- 顶部保留已选数量 pill 与“清空”动作

空状态：

- 明确提示“今日暂无可引用观点，请先前往观点库录入”
- 不展示历史观点作为回退选项

## 6. UI/UX 约束

必须遵循以下现有品牌语言：

- 页面壳统一使用 `DashboardPageShell`
- 概览信息复用 `MetaPillList`
- 主表面继续使用 `rounded-3xl border border-border/60 bg-card/80 shadow-sm`
- 次级容器继续使用 `rounded-2xl`
- 只使用语义色与 token，不写新的业务硬编码品牌色
- 动效继续沿用现有 `animate-in-up-*`
- 历史入口使用次级按钮，避免抢占“添加观点”的主操作权重

不做的事：

- 不新增独立 Hero 区
- 不引入复杂筛选器
- 不在主页面同时平铺历史观点
- 不把历史观点折叠在主页面底部

## 7. 实现边界

本轮实现应覆盖：

1. 数据库同步问题修复路径
2. `GET /viewpoints` 的 scope 过滤
3. 主页面只展示今日观点
4. 新增历史归档页
5. 仿写阶段仅展示今日观点
6. 观点选择区样式优化

本轮不覆盖：

1. 观点编辑能力
2. 历史观点恢复为“今日”的概念
3. 历史观点多维筛选
4. 为观点建立标签、来源、分类体系

## 8. 错误处理

### 8.1 数据库未同步

开发环境若未完成 migration，应优先通过数据库同步解决。

实现层面：

- 不在 repository 中吞掉 `P2021`
- API 继续走统一错误返回
- 可在开发态文案中提供更明确的排查方向，但不伪装成业务错误

### 8.2 空状态

- 今日无观点：显示今日空状态
- 历史无观点：显示归档空状态
- 仿写无今日观点：显示不可引用提示

### 8.3 搜索无结果

- 今日页：明确是“今天没有匹配结果”
- 历史页：明确是“历史归档中没有匹配结果”

## 9. 测试与验证

至少验证以下内容：

### 9.1 后端

- `scope=today` 只返回上海时区当天数据
- `scope=history` 不返回当天数据
- 默认 scope 为 `today`
- cursor 查询在 scope 下仍保持稳定顺序

### 9.2 前端

- `/viewpoints` 仅渲染今日观点
- `/viewpoints/archived` 可查看历史归档
- 主页面历史入口可正常跳转
- 仿写阶段只加载今日观点
- 观点选择卡片的选中/取消/清空行为正常

### 9.3 数据库

- 本地执行 migration 后 `fragments` 表存在
- `POST /api/viewpoints` 可成功创建观点

## 10. 推荐落地顺序

1. 先同步数据库结构，消除 `fragments` 缺表阻塞
2. 改造服务端 scope 查询
3. 改造主页面为“今日观点池”
4. 增加历史归档页
5. 限制仿写只取今日观点
6. 优化仿写区观点选择交互

## 11. 最终决策摘要

- 采用方案 A：主页面仅展示今日观点，历史独立归档页
- 今日/历史按 `Asia/Shanghai` 自然日划分
- 不新增数据库字段，直接基于 `createdAt` 计算
- 仿写阶段只能选择今日观点
- 历史页按日期分组展示
- 仿写区观点选择从 checkbox 列表升级为轻量可点选卡片列表
