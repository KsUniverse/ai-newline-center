# v0.5.0 背景说明 - 成长型 AI 仿写

> 版本: v0.5.0
> 里程碑: v0.5.x（成长型 AI 创作闭环）
> 创建日期: 2026-04-26
> 文档定位: 需求背景、技术调研结论、MVP 边界、阶段路线

---

## 1. 背景

v0.3.3 已经完成 AI 仿写生成的第一阶段能力：用户在对标视频工作台中完成转录和拆解后，选择目标账号、观点素材和模型，即可生成 `RewriteVersion`，并在页面中编辑保存。

这个能力解决了“能生成”的问题，但还没有解决“越用越贴合账号”的问题。当前仿写 prompt 主要依赖本次对标视频、拆解批注、今日观点、目标账号昵称和简介。它不知道过去哪些仿写稿被采用，也不知道采用后的视频表现如何，因此每次生成基本是独立事件。

v0.5.0 的目标是把仿写升级为账号级学习闭环：每次仿写都可以读取该账号已有的仿写案例、原文案拆解、最终发布文案和发布反馈数据，从中形成隐式账号风格记忆，再参与下一次文案生成。

---

## 2. 核心问题

当前系统存在以下断点：

1. `RewriteVersion` 与真实发布视频没有绑定关系，无法知道哪篇仿写稿最终发布到了哪个 `DouyinVideo`。
2. 已有发布数据停留在视频维度，缺少“仿写版本级”的效果归因。
3. AI 生成 prompt 没有注入历史高表现案例和低表现经验。
4. 目标账号风格只来自昵称和签名，缺少由实际采用稿和传播反馈沉淀出来的风格画像。
5. 历史观点素材通过 `usedFragmentIds` JSON 引用，没有生成时上下文快照，复盘能力弱。

v0.5.0 不改变已有 AI 工作台主流程，而是在当前 `Rewrite + RewriteVersion + BullMQ RewriteWorker` 链路上补齐学习数据层、检索层和发布反馈绑定层。

---

## 3. 技术调研结论

### 3.1 适合当前阶段的路线

推荐采用“案例库 RAG + 长期记忆 + 反馈评分”的组合。

成熟做法是把历史输入、输出和反馈沉淀为案例库，通过 embedding 检索相似案例，再让 LLM 基于高质量案例生成新文案。长期记忆用于保存稳定偏好，例如常用结构、语气、开场方式、禁用表达和高表现 CTA。

这种方式适合当前项目，因为：

- 不需要立即微调模型。
- 可以复用现有 MySQL、Prisma、BullMQ、Vercel AI SDK 架构。
- 生成质量会随绑定案例增多而逐步改善。
- 每次生成都有可追溯上下文快照。

### 3.2 暂不优先的路线

微调、DPO、LoRA 和 Bandit 不作为 v0.5.0 首版目标。

原因：

- 微调需要足够多、质量稳定的样本，当前缺少仿写版本与真实反馈的映射。
- DPO 需要成对偏好数据，当前没有持续积累“好稿/差稿”对。
- Bandit 适合已有真实流量后的在线策略探索，当前还缺发布归因和样本池。
- 自动发布涉及平台接口、账号安全和风控，不纳入学习闭环 MVP。

## 4. MVP 边界

### 4.1 范围内

1. 新增 `RewritePublication`，把一个 `RewriteVersion` 手动绑定到目标账号已同步的 `DouyinVideo`。
2. 新增 `RewriteLearningCase`，把已绑定版本沉淀为可检索学习案例。
3. 新增 `DouyinAccountStyleProfile`，按目标账号聚合隐式风格画像。
4. 新增 embedding 生成与检索能力，默认使用 MySQL JSON 存储向量。
5. 预留 Qdrant 适配器，允许部署条件满足后切换独立向量库。
6. 改造 `RewriteWorker` prompt，注入账号画像和相似高表现案例。
7. 前端在版本区域增加“关联已发布视频”入口。
8. 后端基于真实发布指标计算表现分，并用于后续案例检索排序。

### 4.2 范围外

1. 不做自动发布到抖音。
2. 不做微调、DPO、LoRA。
3. 不做 Bandit 在线策略探索。
4. 不做独立账号学习看板。
5. 不在第一版展示完整风格画像，只后台隐式使用。
6. 不从目标账号普通历史视频冷启动学习，只学习已绑定的仿写案例。
7. 不改变每次点击“仿写”生成单稿版本的交互。

---

## 5. 第一版学习闭环

```
对标视频
  -> 转录原文
  -> 拆解标注
  -> 仿写生成 RewriteVersion
  -> 用户编辑最终稿
  -> 用户手动绑定已发布 DouyinVideo
  -> 读取播放/赞评转藏/快照增长
  -> 生成 RewriteLearningCase
  -> 更新 DouyinAccountStyleProfile
  -> 下一次仿写检索并注入
```

第一版中的“人工”只承担两个动作：

1. 将真实发布视频绑定到对应仿写版本。
2. 发现错误绑定时解绑。

AI 学习主要依赖绑定后的真实视频数据。人工采用行为只代表这个版本进入学习样本，不作为长期主评价信号。

---

## 6. 部署策略

当前服务器配置为 2C2GB。独立向量库虽然能力更强，但会引入额外内存、鉴权、备份、监控和升级成本。因此 v0.5.0 采用以下策略：

1. 默认后端为 `mysql`。
2. embedding 存在 `RewriteLearningCase.embeddingJson`。
3. 账号内样本量较小时，在应用层用 cosine similarity 排序。
4. 保留 `VectorStoreAdapter` 抽象。
5. 当服务器资源或外部 Qdrant 服务可用时，通过环境变量切换到 `qdrant`。

## 7. 阶段路线

### v0.5.0 - 学习闭环 MVP

- 版本到发布视频的手动绑定。
- 学习案例入库。
- MySQL embedding 检索。
- 账号画像后台生成。
- 生成 prompt 注入历史案例和画像。

### v0.5.1 - 归因体验增强

- 系统按目标账号、发布时间、文案相似度给出候选视频。
- 用户确认后绑定，减少手动查找成本。
- 增加绑定错误检测，例如账号不匹配、发布时间异常。

### v0.5.2 - 评估和提示词优化

- 增加 LLM-as-judge 离线评分。
- 建立 eval 数据集。
- 用历史案例评估 prompt 版本。
- 具备稳定样本后评估 DSPy 自动提示词优化。

### v0.6.x - 自动策略优化

- 基于真实反馈做候选排序。
- 引入 Bandit 优化标题、开头、CTA、结构策略。
- 数据量充分后再评估微调或偏好学习。

---

## 8. 关键术语

- `Rewrite`：一个 AI 工作台下的仿写任务容器。
- `RewriteVersion`：每次点击“仿写”产生的一个版本。
- `RewritePublication`：某个仿写版本与真实发布视频的绑定记录。
- `RewriteLearningCase`：可被检索和学习的历史仿写案例。
- `DouyinAccountStyleProfile`：某个目标账号的隐式风格画像。
- `VectorStoreAdapter`：向量检索适配层，默认 MySQL，可选 Qdrant。
- `learningContextSnapshot`：某次生成使用到的画像和案例快照。
