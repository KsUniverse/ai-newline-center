## 摘要

- 版本目标：为员工抖音账号录入补齐扫码登录、账号级登录态持久化和收藏同步 Cookie 传递能力。
- 正式交付：网页扫码自动建号、账号重登录、登录状态展示、多账号隔离、收藏同步读取账号级 Cookie header。
- 身份识别口径：登录成功后只认抖音网页侧触发的 aweme/favorite 请求。
- 运行前提：Playwright 和 Chromium 由服务端直接运行，登录态落到服务端私有目录。

---

## 收口结论

1. 本版只维护一条主链路：扫码成功后，从 aweme/favorite 请求中提取 sec_user_id 和 cookie。
2. 若登录成功后没有捕获到 aweme/favorite 请求，则会话直接失败，不自动建号，不绑定正式登录态。
3. 收藏同步调用 crawler 时只传当前账号绑定的原始 Cookie header 字符串，不传 storageState 文件。
4. 任一异常场景都必须 fail-closed：不误建号、不误绑登录态、不串号。

## 功能清单

### Feature-2A：员工抖音账号扫码登录与自动建号 (P0)

描述：员工在系统内新增抖音账号时，通过网页弹窗完成抖音扫码登录。登录成功后，系统从 aweme/favorite 请求中识别 secUserId，识别成功则自动创建 MY_ACCOUNT 并绑定该账号的正式登录态。

交互：
1. 员工点击“新增账号”。
2. 页面弹窗展示二维码，并通过轮询更新状态。
3. 员工使用抖音 App 扫码并在手机端确认。
4. 登录成功后，系统自动识别账号身份并创建账号。
5. 创建成功后，页面展示新增账号及其登录状态。

运行前提：
- Playwright / Chromium 由服务端直接运行。
- 服务器或镜像允许安装并运行 Chromium。
- 服务端提供私有、可写的登录态文件目录。
- 员工仅通过浏览器访问系统，不依赖本地额外环境。

边界：
- 登录成功但未能捕获 aweme/favorite 请求时，不创建账号，不保留临时登录态。
- 一个员工连续录入多个抖音账号时，每次扫码会话必须独立，不能复用上一账号的登录态。
- 二维码失效、扫码超时、手机端取消确认时，页面需给出明确状态并允许重新发起。

验收标准：
1. 员工点击“新增账号”后，可在系统网页内看到二维码，并通过轮询感知“待扫码 / 已扫码待确认 / 登录成功 / 失败 / 已失效”等状态变化。
2. 登录成功且成功识别 secUserId 时，系统自动创建一条 MY_ACCOUNT。
3. 账号创建成功后，系统为该账号保存一份独立的正式登录态，且后续可识别该账号当前登录状态。
4. 同一员工连续新增两个不同抖音账号时，系统生成两条独立账号记录与两份独立登录态，不发生串号。
5. 若未捕获到 aweme/favorite 请求，系统不创建账号、不误绑定登录态。

### Feature-2B：登录态持久化、状态展示与收藏同步 Cookie 传递 (P0)

描述：系统按抖音账号维度持久化登录态，并将其用于收藏同步。员工可查看每个账号的登录状态，在失效后重新登录；收藏同步调用 crawler 收藏接口时，使用当前账号绑定的原始 Cookie header 字符串。

交互：
1. 员工在账号列表或详情中查看账号登录状态。
2. 对已录入账号可执行“重新登录”。
3. 登录失效后，页面展示“已失效”并允许重新扫码。
4. 后台收藏同步按账号读取对应登录态并发起抓取。

数据：
- 登录状态：NOT_LOGGED_IN / PENDING / LOGGED_IN / EXPIRED / FAILED。
- 数据库存登录状态元信息。
- 服务端私有目录中的账号登录态文件。

外部对接口径：
- 收藏同步调用 crawler 收藏接口时，由本服务直接读取账号上绑定的 aweme/favorite 请求头原始 Cookie header 字符串。
- 本服务向 crawler 传递的是 Cookie header 字符串，而不是 storageState 文件。
- Cookie 的读取、失效判断和重新绑定均以当前账号维度处理，不按员工维度共用。

验收标准：
1. 每个已录入抖音账号都能独立展示登录状态，并支持在失效后重新登录。
2. 收藏同步发起 crawler 收藏接口调用时，使用的是当前账号绑定的原始 Cookie header 字符串。
3. crawler 收藏接口不接收 storageState 文件，只接收本服务传递的 Cookie header 字符串。
4. 同一员工下多个账号的登录态彼此隔离，不因重登、失效或同步而互相污染。
5. 系统在登录态不可用时，不误抓取、不误绑定，并明确要求员工重新登录。

## 本版不包含

| 内容 | 说明 |
|------|------|
| 多来源身份兜底解析 | 不回退到 info 接口、DOM、header 或 bodyText 解析 |
| 依赖客户端本地环境完成登录 | 统一由服务端直接运行 Playwright / Chromium |
| 把 storageState 文件直接传给 crawler | crawler 只接收 Cookie header 字符串 |

## 本轮新增问题记录（未在本轮代码实现）

### Issue-Sync-01：账号信息同步 / 视频同步未体现定时更新结果

现象：
- 账号详情页中的“最后同步时间”停留在用户较早前手动同步的时间。
- 下方视频列表中的互动数据长期不变。
- 数据库中该账号的 lastSyncedAt 未体现按小时推进。
- 数据库中该账号下视频记录的 updatedAt 仍停留在初次入库时间附近。

已确认事实：
1. 当前环境未配置 ACCOUNT_SYNC_CRON / VIDEO_SYNC_CRON，理论上应走默认规则。
2. 账号信息同步与视频同步逻辑本身当前都不依赖登录态。
3. 因此，即使账号当前显示“未登录”，也不能直接解释“个人信息同步 / 视频同步一直没有变化”。

当前判断：
- 更像是定时器在当前运行环境下没有稳定生效，或同步链路全部失败但缺乏可见运行日志。

本轮不处理原因：
- 本轮优先级聚焦扫码登录、登录态持久化和自动建号。
- 当前抖音登录链路仍受风控影响，继续扩大改动面会显著增加联调噪音。
