## 问题清单

---

### Bug-1: 收藏同步未触发对标博主新增 (P0)

**问题类型**: Bug

**问题描述**  
员工在抖音收藏了新博主的视频后，系统日志未见对应 BENCHMARK_ACCOUNT 被创建，对标账号列表中也无新增记录。收藏同步定时器（第四个定时器，每 5 分钟）疑似未正确执行「识别博主 secUserId → 查 DB → 创建 BENCHMARK_ACCOUNT」这一核心链路。

**背景（来自 v0.2.2 设计）**  
收藏同步流程规定：
1. 遍历员工 MY_ACCOUNT 的收藏视频，提取所属博主 `secUserId`
2. 查 DB，若该 secUserId 不存在 BENCHMARK_ACCOUNT（非归档、非已有）→ 调用 `fetchUserProfile` → 创建

**期望行为**  
- 员工在抖音收藏了某博主的视频，且收藏时间在近 1 小时内
- 系统在 5 分钟内完成以下操作：
  1. 遍历该员工的收藏列表，识别视频所属博主 `secUserId`
  2. 查 DB，确认未存在对应 BENCHMARK_ACCOUNT
  3. 调用 `fetchUserProfile(secUserId)` 获取博主信息
  4. 创建 `DouyinAccount(type=BENCHMARK_ACCOUNT)`，并写入 DB
- 创建完成后，对标账号列表中出现该博主卡片

**验收标准**  
1. 收藏同步定时器运行后，日志可见「收藏同步开始」「扫描账号 X 条」「发现新博主 secUserId=xxx」「创建 BENCHMARK_ACCOUNT 成功」等关键节点日志
2. 员工收藏新博主视频（收藏时间在 1 小时内），下次定时器执行后，对标账号列表中出现该博主
3. 同一博主被多个员工收藏时，DB 中只有一条 BENCHMARK_ACCOUNT 记录（secUserId 全局唯一）
4. 爬虫调用失败（单个账号）时，仅跳过该账号，其余账号继续处理，整体任务不中断

### Bug-2: 视频同步缺少重复校验导致封面重复下载 (P0)

**问题类型**: Bug

**问题描述**  
日志显示「待下载封面」，但手动触发同步后，实际并无新增短视频入库。疑似视频同步逻辑在入库前缺少对 `videoId`（awemeId）的去重校验，导致已有视频被重复处理、封面被重复下载，浪费 I/O 资源，且日志误导判断。

**期望行为**  
- 每次视频同步定时器执行时：
  1. 拉取账号的视频列表（首次同步取前 10 条；增量同步取最新 4 条）
  2. **入库前**先查询 DB，检查该 `videoId`（awemeId）是否已存在
  3. 已存在 → 更新已有记录的数据指标（播放数、点赞数等），**不重新下载封面**
  4. 不存在 → 新建视频记录，触发封面和视频文件下载
- 「待下载封面」日志仅在真正需要下载时出现（新视频），不对已有视频重复输出

**验收标准**  
1. 对已有视频执行同步时，DB 中对应记录的 `updatedAt` 更新，但 `coverStoragePath` 不发生变化（文件未被重复下载）
2. 「待下载封面」日志数量 = 本次同步新增视频数量（不含已有视频）
3. 完整执行一轮视频同步后，DB 中无因重复处理产生的重复 `videoId` 记录
4. 新发布的视频（DB 中不存在）仍正常新建、正常触发封面下载

### Bug-3: 对标账号未纳入视频同步流程（依赖 Bug-1 修复后验证） (P0)

**问题类型**: Bug（需 Bug-1 修复并有 BENCHMARK_ACCOUNT 数据后才可验证）

**问题描述**  
Bug-1 修复后，系统可正常创建 BENCHMARK_ACCOUNT，但尚未确认这些对标账号是否被 F-001-2 的视频同步定时器正确扫描。  
按 PRD F-001-2 设计，视频同步定时器应扫描**所有** DouyinAccount（含 MY_ACCOUNT 和 BENCHMARK_ACCOUNT），无 type 过滤。若实现中存在 type 过滤硬编码，则对标账号视频将永远不会同步。

**期望行为**  
- 视频同步定时器（默认每 1 小时）扫描范围：所有 `deletedAt IS NULL` 的 DouyinAccount，**不区分 type**
- 对标账号在被创建后，最迟 1 小时内被视频同步定时器抓取到视频列表，对标账号详情页展示视频

**验收标准**  
1. 视频同步定时器日志中可见 BENCHMARK_ACCOUNT 类型的账号被扫描（不仅限于 MY_ACCOUNT）
2. Bug-1 修复后新创建的 BENCHMARK_ACCOUNT，在视频同步定时器下次执行后，其详情页出现视频列表
3. 若现有代码存在 `type = MY_ACCOUNT` 的硬编码过滤，须移除该过滤条件

### Feature-1: 涉及定时器的页面增加前端自动刷新 + Loading 标志 (P1)

**问题类型**: 功能增强

**背景**  
系统存在多个后台定时任务（账号同步、视频同步、收藏同步等），但前端页面数据为一次性加载，用户无法感知到后台任务的进展，需手动刷新页面才能看到最新状态。

**涉及页面范围**（按优先级）：
| 页面 | 路径 | 自动刷新间隔 |
|------|------|------------|
| 我的账号列表 | `/my-accounts` 或 `/accounts` | 60 秒 |
| 对标账号列表 | `/benchmarks` | 60 秒 |
| 对标账号详情（视频列表） | `/benchmarks/[id]` | 60 秒 |
| 我的账号详情（视频列表） | `/accounts/[id]` 或类似路径 | 60 秒 |

> 上述路径以实际代码实现为准，不强制要求与此一致；自动刷新间隔建议固定 60 秒，不可配置（前端常量即可）。

**期望行为**

**自动刷新**：
- 页面挂载后，每隔 60 秒自动重新请求当前页面的核心数据（账号列表或视频列表）
- 用户操作（如打开 Drawer、输入表单）期间，自动刷新不打断用户（数据静默更新，不重置 UI 状态和滚动位置）
- 页面卸载/切换时，自动刷新定时器自动清除，不留后台任务

**Loading 标志**：
- 在页面右上角或列表标题区域放置一个**低调的旋转小图标**（小于主操作按钮的 icon 尺寸），始终可见（不仅在刷新时出现）
- 自动刷新触发时，图标进入旋转动画；刷新完成后，动画停止（保持图标静止可见）
- 图标旁可选显示「N秒后刷新」或「最后刷新：HH:mm:ss」文字提示（不强制要求，UI 设计自由裁量）
- Loading 标志**不影响**主操作区域的布局和交互

**边界**：
- 自动刷新仅针对列表数据（账号卡片列表、视频列表），不刷新整个页面（无 `location.reload()`）
- 网络请求失败时，自动刷新跳过本次，不中断后续定时，不向用户弹出错误提示
- Loading 标志风格与 Linear 暗色主题一致，颜色低调（不使用醒目的主色调），不喧宾夺主

**验收标准**  
1. 打开账号列表页或对标账号列表页，60 秒后页面数据自动刷新，若后台有新数据，列表自动出现新卡片（无需手动刷新浏览器）
2. 自动刷新时，Loading 小图标进入旋转动画；完成后动画停止
3. 页面切换（路由跳转）后，旧页面的自动刷新定时器不再执行（无内存泄漏或后台请求）
4. 自动刷新期间，Drawer/Dialog 处于打开状态时，刷新完成后 Drawer/Dialog 保持打开（UI 状态不被重置）

## 依赖与约束

- **Bug-3 依赖 Bug-1**：需先确认收藏同步能正常创建 BENCHMARK_ACCOUNT，才能验证视频同步是否覆盖对标账号
- **不新增定时器**：Feature-1 的自动刷新为纯前端客户端 `setInterval`，不涉及后端改动
- **不修改 PRD 模块定义**：本迭代所有修复均在 F-001-2 / F-002-1 现有设计范围内，无新模块新字段
