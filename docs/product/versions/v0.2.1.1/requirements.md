## 用户故事

**作为开发者**，我希望平台能对接真实的爬虫 API 获取抖音数据，  
**以便**所有账号信息和视频数据来自抖音真实内容而非 mock 数据。

**作为运维**，我希望所有爬虫 API 响应都有完整的 JSON 日志，  
**以便**在字段映射不准确时可根据日志快速定位并指导修复。

**作为员工**，我希望每条视频的播放数据以 10 分钟粒度被持续采集，  
**以便**后续查看数据趋势图时有足够精细的时间线。

**作为员工**，我希望在账号首页能一目了然看到所有账号的视频汇总，  
**以便**快速浏览和筛选我管理的所有抖音内容，而不需要逐个点进账号查看。

## 功能清单

### F-CRAWLER-1: CrawlerService 重写 — 真实 API 对接 (P0)

**描述**: 将现有 CrawlerService 中的 mock 实现替换为真实爬虫 REST API 调用。

**爬虫服务信息**:
- 基础地址: 环境变量 `CRAWLER_API_URL`（开发默认 `http://localhost:8011`）
- 协议: HTTP REST / JSON
- 接口文档: `{CRAWLER_API_URL}/docs`

**需对接的 5 个接口**:

#### 接口 1: URL → secUserId

| 项目 | 值 |
|------|----|
| **路径** | `/api/douyin/web/get_sec_user_id` |
| **用途** | 将抖音主页 URL 转换为 secUserId |
| **入参** | `url`: 抖音主页链接（如 `https://www.douyin.com/user/xxx`） |
| **出参** | `sec_user_id`: 字符串 |
| **调用时机** | 添加账号时（F-001-1 流程第一步） |

#### 接口 2: 获取账号资料

| 项目 | 值 |
|------|----|
| **路径** | `/api/douyin/web/handler_user_profile` |
| **用途** | 根据 secUserId 获取账号详细资料 |
| **入参** | `sec_user_id` |
| **出参** | 昵称、头像、粉丝数、作品数、简介等 |
| **调用时机** | 添加账号时（第二步），以及账号信息同步定时器 |

#### 接口 3: 获取视频列表

| 项目 | 值 |
|------|----|
| **路径** | 通过 secUserId 获取（具体路径从爬虫文档确认） |
| **用途** | 获取账号下的视频列表 |
| **入参** | `sec_user_id`, 分页参数 |
| **出参** | 视频列表（视频ID、标题、封面、数据指标等） |
| **调用时机** | 视频同步定时器 |

#### 接口 4: 获取收藏视频

| 项目 | 值 |
|------|----|
| **路径** | `/api/douyin/web/fetch_user_collection_videos` |
| **用途** | 获取用户收藏的视频列表 |
| **入参** | `sec_user_id` |
| **出参** | 收藏视频列表及所属博主信息 |
| **调用时机** | v0.2.2 收藏同步使用（本版本仅实现接口封装 + 日志，不调用） |

#### 接口 5: 获取单条视频详情

| 项目 | 值 |
|------|----|
| **路径** | `/api/douyin/web/fetch_one_video` |
| **用途** | 获取单条视频的最新播放数据 |
| **入参** | `aweme_id`（视频ID） |
| **出参** | 视频最新播放量、点赞数、评论数、转发数 |
| **调用时机** | VideoSnapshot 采集定时器（每 10 分钟） |

**日志策略**:
- 所有爬虫接口响应必须以 `logger.info` 完整打印原始 JSON 结构体
- 日志格式: `[CrawlerService] <接口名> response: <完整 JSON>`
- 目的: 开发者检查日志后逐步完善字段映射，不需要一次性完美对接
- 字段映射首次以合理猜测实现，后续根据日志对齐

**边界**:
- 爬虫服务不可用时，调用方需正确处理异常（与现有 mock 的错误处理行为一致）
- 所有接口调用需设置合理超时（建议 30s）
- 保持 CrawlerService 的接口签名和返回类型不变（内部实现替换，外部行为兼容）

### F-CRAWLER-2: secUserId 引入 (P0)

**描述**: 在 DouyinAccount 数据模型中新增 `secUserId` 字段，作为后续所有爬虫接口的核心标识。

**Schema 变更**:
- `DouyinAccount` 新增字段: `secUserId String @unique`（抖音用户安全ID）
- 该字段在添加账号时（第一步 URL → secUserId）获取并持久化
- 后续所有爬虫操作（账号信息同步、视频列表同步、收藏同步）均通过 secUserId 调用

**添加账号流程变更**（F-001-1 升级）:
1. 用户输入抖音主页 URL
2. **新增**: 调用接口 1 (`get_sec_user_id`) 获取 `secUserId`
3. 调用接口 2 (`handler_user_profile`) 获取账号信息
4. 创建 DouyinAccount 记录（含 `secUserId`）

**同步流程变更**:
- 账号信息同步: 以 `secUserId` 为入参调用接口 2（替换原 `profileUrl`）
- 视频同步: 以 `secUserId` 为入参调用接口 3（替换原 `profileUrl`）

**边界**:
- 历史已有账号（secUserId 为空）: 首次同步时从 `profileUrl` 补充获取 `secUserId`
- secUserId 全局唯一，作为去重依据之一

### F-CRAWLER-3: 视频同步策略升级 (P0)

**描述**: 将现有视频同步策略从固定页数翻页升级为智能增量策略。

**新策略**:
- **首次同步**（账号下无任何视频记录）: 取前 10 条视频
- **增量同步**（账号下已有视频记录）: 从最新视频开始取，每次取 4 条，若 4 条中有 DB 已存在的 videoId 则停止继续翻页；若 4 条全部为新视频则继续取下一批 4 条，直到遇到已存在的 videoId 或达到安全上限

**策略优势**: 大幅减少无效抓取，仅拉取真正的新视频

**边界**:
- 安全上限：增量同步最多翻页 N 次（建议 10 次，即最多 40 条新视频），防止异常情况下无限翻页
- 增量停止判断: 遇到 DB 已存在的 videoId 即停止，无需按发布时间判断

### F-CRAWLER-4: VideoSnapshot 快照模型 + 定时器 (P0)

**描述**: 新增 VideoSnapshot 数据模型和对应的 10 分钟采集定时器，为后续数据趋势展示做数据积累。

**Schema 新增**:

```
model VideoSnapshot {
  id        String   @id @default(cuid())
  videoId   String   // 关联 DouyinVideo.id
  timestamp DateTime @default(now())  // 采集时间
  playsCount    Int
  likesCount    Int
  commentsCount Int
  sharesCount   Int
}
```

**定时器**:
- 名称: 视频快照采集定时器
- 频率: 每 10 分钟（可配置，环境变量 `VIDEO_SNAPSHOT_CRON`）
- 执行逻辑:
  1. 从 DB 获取所有 DouyinVideo
  2. 逐条调用接口 5 (`fetch_one_video`)，获取最新播放数据
  3. 写入 VideoSnapshot 记录（不覆盖历史，每次采集新增一条）
  4. 同时更新 DouyinVideo 上的数据指标字段（保持与 v0.2.1 行为兼容）
- 失败处理: 单条视频失败跳过，不中断整批

**边界**:
- 视频数量增长后需考虑采集耗时（当前阶段按顺序逐条采集，后续可优化）
- VideoSnapshot 历史数据不自动清理（后续按需添加清理策略）

### F-CRAWLER-5: 文件存储骨架 (P0)

**描述**: 建立文件下载和存储的基础服务骨架，为后续视频封面和视频文件本地存储做准备。

**StorageService 骨架**:
- `downloadAndStore(url: string, category: string): Promise<string>` — 下载远程文件到本地，返回本地路径
- 存储路径: `storage/{category}/{date}/{filename}`
- 当前阶段: 仅实现骨架接口 + 日志（打印待下载的 URL），不实际执行下载
- 后续迭代: 接入实际下载逻辑，再后续迁移 OSS

**调用时机**: 暂不实际调用。在 CrawlerService 返回 coverUrl / videoUrl 时打印日志标记「待下载」

**边界**:
- 不影响现有功能，纯基础设施预留
- 接口设计需考虑后续 OSS 迁移的兼容性

### F-HOME-1: 账号首页改版 — 跨账号视频网格 (P0)

**描述**: 改版「我的账号」首页布局。顶部横向展示账号卡片行（最多约 4 个），下方大面积展示所有账号的视频汇总网格，支持筛选和排序。

**页面布局**:

```
┌─────────────────────────────────────────────────────────┐
│ 标题 + 添加按钮                                          │
├───────────┬───────────┬───────────┬───────────┬─────────┤
│ 账号卡片1  │ 账号卡片2  │ 账号卡片3  │ 账号卡片4  │(+添加)  │
├───────────┴───────────┴───────────┴───────────┴─────────┤
│ [筛选栏] 账号筛选 | 标签筛选 | 排序(时间/点赞量)          │
├──────────┬──────────┬──────────┬─────────────────────────┤
│ 视频卡片  │ 视频卡片  │ 视频卡片  │ ...                    │
│  (3:4)   │  (3:4)   │  (3:4)   │                        │
│ ♡ 6028   │ ♡ 1896   │ ♡ 3639   │                        │
│ 标题...   │ 标题...   │ 标题...   │                        │
│ 时间      │ 时间      │ 时间      │                        │
└──────────┴──────────┴──────────┴─────────────────────────┘
```

**顶部账号行**:
- 复用现有 `AccountCard` 组件（或精简版），横向排列
- 点击账号卡片跳转到账号详情页（保持现有行为）
- 员工角色显示「添加」按钮

**视频网格**:
- 数据范围: 当前用户权限范围内所有账号的全部视频（跨账号汇总）
- 默认排序: 按发布时间倒序
- 网格布局: 响应式多列（参考抖音风格，3-4 列）
- 分页或无限滚动

**视频卡片设计（3:4 比例）**:
- 封面图铺满卡片（object-cover）
- 底部半透明渐变遮罩，显示:
  - 标题（单行截断）
  - 左侧: ♡ 图标 + 点赞数
  - 右侧: 发布时间（格式 `yyyy-MM-dd HH:mm`）
- 如有标签，在卡片上方或底部显示标签 badge

**悬浮播放交互**:
- 默认显示封面图（静态）
- 鼠标 hover 时: 自动播放视频（如有 videoUrl），封面图淡出
- 鼠标离开时: 暂停播放，回到封面图
- 无 videoUrl 时: hover 无播放效果，仅显示封面

**筛选与排序**:

| 功能 | 说明 |
|------|------|
| 账号筛选 | 下拉选择某个账号，仅显示该账号的视频；默认「全部账号」 |
| 标签筛选 | 下拉选择某个标签，仅显示含该标签的视频；默认「全部标签」 |
| 排序切换 | 可选「最新发布」（默认）或「最多点赞」 |

**Schema 变更 — 视频标签**:
- `DouyinVideo` 新增字段: `tags String[]`（标签数组）
- 标签数据如何写入（自动提取 or 手动标注）本版本不实现，字段预留为空数组
- 前端筛选器从已有视频的 tags 去重生成可选项

**API 变更**:
- 新增跨账号视频列表端点: `GET /api/videos`
  - Query 参数: `page`, `limit`, `accountId?`（筛选）, `tag?`（筛选）, `sort=publishedAt|likeCount`（排序）, `order=desc`
  - 返回: `PaginatedData<DouyinVideoDTO>`（附带 `accountNickname` 或关联信息）
  - 权限: 自动按 session 用户角色过滤数据（员工仅自己、负责人本公司、管理员全部）

**边界**:
- 视频总量可能较多，必须分页
- 标签为空时，标签筛选器不显示（或显示但无可选项）
- 无视频时显示空状态
- hover 播放需考虑性能（同时只有一个视频在播放）

## 验收标准

### 爬虫对接

1. CrawlerService 调用真实爬虫 API（非 mock），能正确获取抖音账号信息和视频列表
2. 所有爬虫接口响应都有完整 JSON 日志输出，可在控制台查看原始返回结构
3. 爬虫服务不可用时，错误处理行为与之前一致（不中断批处理、不覆盖已有数据）

### secUserId

4. 添加账号流程经过两步（URL → secUserId → 账号信息），secUserId 持久化到 DB
5. 账号信息同步和视频同步均使用 secUserId 作为入参（不再依赖 profileUrl）
6. 已有账号（secUserId 为空）在首次同步时自动补充

### 视频同步策略

7. 首次同步取前 10 条视频
8. 增量同步取最新 4 条/批，遇到已存在 videoId 自动停止
9. 增量同步有安全翻页上限

### VideoSnapshot

10. VideoSnapshot 定时器每 10 分钟运行，逐条采集视频最新数据
11. 每次采集写入新的 VideoSnapshot 记录（不覆盖历史）
12. DouyinVideo 上的数据指标字段同步更新

### 文件存储

13. StorageService 骨架已创建，接口定义完整
14. 封面/视频 URL 在日志中有「待下载」标记

### 账号首页改版

15. 账号首页顶部显示账号卡片行，下方为视频网格
16. 视频网格默认按发布时间倒序展示所有账号的视频
17. 可按账号筛选视频
18. 可按标签筛选视频（标签为空时筛选器无选项或隐藏）
19. 可切换排序方式: 最新发布 / 最多点赞
20. 视频卡片为 3:4 比例，显示封面、标题、点赞数、发布时间
21. 鼠标 hover 视频卡片时自动播放视频，离开时暂停回到封面
22. 发布时间格式为 `yyyy-MM-dd HH:mm`
23. 点击视频卡片可查看视频详情
