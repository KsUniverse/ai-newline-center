# 产品需求规格书 (PRD) — AI Newline Center

> 本文档为完整产品需求规格书，是所有版本迭代的全局参考。各版本 requirements.md 从本文档中按范围摘取。

## 摘要

- **产品定位**: 面向集团/分公司的 AI 短视频文案生产平台
- **核心价值**: 将"找爆款→拆解→仿写→发布→复盘"全链路 AI 化，让 AI 越来越懂每个账号的内容风格
- **用户角色**: 超级管理员 / 分公司负责人 / 员工（共 3 种）
- **功能模块**: 7 个（M-000 ~ M-006）
- **优先级分布**: P0(7) — 全部为核心必做

## 模块总览

| 模块 | 名称 | 定位 | 建议版本 |
|------|------|------|----------|
| M-000 | 组织与权限 | 地基 — 登录、组织、权限体系 | v0.1 |
| M-001 | 我的账号 | 产出端 — 员工管理的抖音号 | v0.2 |
| M-002 | 对标账号 | 输入端 — 爬取收藏发现爆款 | v0.2 |
| M-003 | AI 拆解 | AI 核心 — 转录+划句批注拆解 | v0.3 |
| M-004 | 碎片观点 | 素材池 — 全公司共享观点 | v0.3 |
| M-005 | AI 仿写 | 核心产出 — 框架+观点→文案 | v0.4 |
| M-006 | 发布复盘 | 闭环 — 数据趋势+经验反馈 | v0.5 |

## M-000: 组织与权限

### 概述

三层组织体系：集团→分公司→员工。分公司级数据隔离。账号密码登录，管理员创建账号。

### 角色与权限

| 角色 | 组织管理 | 账号管理 | 内容操作 | 数据查看 | 系统配置 |
|------|---------|---------|---------|---------|---------|
| **超级管理员** | 创建/管理集团和分公司 | 创建/管理所有用户账号 | — | 全部数据 | AI模型配置、Prompt模板管理 |
| **分公司负责人** | 管理本公司员工 | 创建本公司员工账号 | — | 本公司全部数据 | — |
| **员工** | — | 管理自己的抖音号 | 全生命周期操作 | 仅自己的数据 | — |

### F-000-1: 用户登录 (P0)

- **描述**: 用户通过账号密码登录平台
- **交互**: 登录页 → 输入账号密码 → 验证 → 进入主界面
- **数据**: User (账号、密码hash、角色、归属组织)
- **边界**:
  - 密码错误提示（不泄露账号是否存在）
  - 登录态有效期管理（JWT/Session）
- **验收标准**:
  1. 管理员创建的账号可正常登录
  2. 不同角色登录后看到对应权限的界面
  3. 密码错误时有明确提示

### F-000-2: 组织管理 (P0)

- **描述**: 超级管理员管理集团和分公司结构
- **交互**: 管理后台 → 组织列表 → 创建/编辑分公司
- **数据**: Organization (名称、类型[集团/分公司]、上级组织)
- **边界**:
  - 分公司必须归属集团
  - 删除分公司需处理其下员工和数据
- **验收标准**:
  1. 可创建分公司并设置负责人
  2. 组织树状结构正确展示

### F-000-3: 用户管理 (P0)

- **描述**: 管理员/负责人创建和管理用户
- **交互**: 用户列表 → 创建用户（指定角色、归属组织）→ 编辑/禁用
- **数据**: User (姓名、账号、角色、归属组织ID、状态)
- **边界**:
  - 分公司负责人只能管理本公司员工
  - 员工归属分公司或集团
  - 禁用账号立即失效
- **验收标准**:
  1. 管理员可创建所有角色的用户
  2. 分公司负责人只能创建/查看本公司员工
  3. 数据按组织隔离

## M-001: 我的账号

### 概述

员工上传自己管理的抖音主页链接，平台通过爬虫定时同步账号信息和视频列表+数据。这是产出端——员工在此发布视频、关联仿写稿、查看数据反馈。

### F-001-1: 添加抖音账号 (P0)

- **描述**: 员工添加自己管理的抖音账号
- **交互**: 我的账号页 → 点击添加 → 输入抖音主页链接 → 提交 → 调用爬虫获取账号基本信息 → 展示确认
- **数据**: DouyinAccount (主页链接、secUserId、账号昵称、头像、粉丝数、作品数、简介、归属员工ID、类型=MY_ACCOUNT)
- **爬虫对接（两步流程）**:
  1. **URL → secUserId**: 调用 `/api/douyin/web/get_sec_user_id`，传入 profileUrl，获取 `sec_user_id`
  2. **secUserId → 账号信息**: 调用 `/api/douyin/web/handler_user_profile`，传入 sec_user_id，获取昵称、头像、粉丝数、作品数、简介等
  - secUserId 是后续所有账号操作（视频列表、收藏列表等）的核心标识，必须持久化存储
- **边界**:
  - 链接格式校验
  - 重复添加检测（同一 profileUrl 全局唯一，一个抖音号只能有一个管理者）
  - 爬虫失败时提示用户
  - 一个员工可添加多个账号
- **验收标准**:
  1. 输入合法链接后自动爬取并展示账号信息
  2. 账号出现在"我的账号"列表中

### F-001-2: 账号信息同步 (P0)

- **描述**: 通过全局定时任务扫描所有账号，分别同步账号基础信息和视频列表
- **交互**: 系统定时任务自动执行；用户可在账号详情页手动触发同步
- **数据**: DouyinAccount (更新字段) + DouyinVideo (视频ID、标题、封面、发布时间、播放量、点赞数、评论数、转发数、视频链接)
- **定时器拆分**: 本功能包含三个独立定时器，各自配置运行间隔：
  - **账号信息同步定时器**: 全局扫描所有 DouyinAccount（含 MY_ACCOUNT 和 BENCHMARK_ACCOUNT），更新粉丝数、作品数等基础信息。频率较低（默认 6 小时）
  - **视频同步定时器**: 全局扫描所有 DouyinAccount，增量抓取新发布视频 + 更新已有视频的数据指标。频率较高（默认 1 小时）
  - **视频快照定时器**: 全局扫描所有 DouyinVideo，通过 `/api/douyin/web/fetch_one_video` 逐条采集最新播放数据，写入 VideoSnapshot 记录。频率最高（默认 10 分钟）
- **爬虫对接**:
  - **账号信息同步**: 调用 `/api/douyin/web/handler_user_profile`（secUserId），更新账号基础信息
  - **视频列表同步**: 通过 secUserId 获取视频列表。首次同步取前 10 条；后续增量同步取最新 4 条，遇到 DB 已存在的 videoId 即停止
  - **视频快照采集**: 调用 `/api/douyin/web/fetch_one_video`（aweme_id），获取单条视频最新数据，写入 VideoSnapshot
- **文件存储**: 视频封面图和视频文件需下载到本地存储（当前阶段本地文件系统，后续迁移 OSS）
- **调度模式**: 全局定时任务按配置间隔扫描所有待同步账号，非按账号独立注册定时器
- **边界**:
  - 同步频率由系统配置（每个定时器的间隔独立设置）
  - 增量同步（只抓取新视频 + 更新已有视频数据）
  - 爬虫失败不影响已有数据
- **验收标准**:
  1. 三个定时任务分别按各自配置间隔自动执行
  2. 新发布的视频自动出现在列表中
  3. 已有视频的数据指标更新
  4. VideoSnapshot 每 10 分钟采集一次，支持趋势图展示

### F-001-3: 账号与视频列表展示 (P0)

- **描述**: 展示员工管理的所有抖音账号及其视频
- **交互**: 我的账号页 → 账号卡片列表（显示头像、昵称、粉丝数、作品数）→ 点击进入 → 视频列表（缩略图、标题、数据指标）
- **数据**: DouyinAccount + DouyinVideo[]
- **边界**:
  - 空状态提示（无账号 / 无视频）
  - 视频列表分页或虚拟滚动
- **验收标准**:
  1. 账号卡片展示关键信息
  2. 视频列表按发布时间倒序
  3. 可查看单个视频详情

## M-002: 对标账号

### 概述

员工在抖音 App 收藏视频后，平台定时爬取收藏列表，自动发现对标博主。这是输入端——提供爆款内容作为 AI 拆解素材。数据结构与"我的账号"几乎一致（复用 DouyinAccount + DouyinVideo），但 type 为 BENCHMARK_ACCOUNT，业务上独立管理。

**对标账号触发链规则**:
- 收藏同步定时器扫描员工的 **MY_ACCOUNT** 账号收藏列表 → 识别视频所属博主 → 自动创建对标账号（`type=BENCHMARK_ACCOUNT`）
- 对标账号被创建后，自动纳入 F-001-2 的全局扫描范围（账号信息同步 + 视频同步）
- **不递归触发收藏同步**：对标账号只同步其账号信息和视频，不爬取其收藏列表，避免无限扩散
- 所有定时器均采用**全局扫描模式**（按配置间隔扫描所有符合条件的账号），非按账号独立注册

### F-002-1: 收藏同步 (P0)

- **描述**: 平台定时爬取员工抖音账号的收藏列表，自动发现新收藏的视频
- **交互**: 系统自动执行。员工在"对标账号"页面看到自动同步的结果
- **数据**: 从员工的 MY_ACCOUNT 抖音号获取收藏列表 → 识别视频 → 自动创建对标账号 (DouyinAccount, 类型=BENCHMARK_ACCOUNT)
- **爬虫对接**: 调用 `/api/douyin/web/fetch_user_collection_videos`，传入 secUserId，获取收藏视频列表及其所属博主信息
- **触发频率**: 每 5 分钟触发一次，扫描所有 `type=MY_ACCOUNT` 且 `secUserId != null` 且 `deletedAt IS NULL` 的账号；作为第四个独立定时器注册，运行间隔独立配置
- **时间窗口增量策略**: 从最新收藏向历史方向遍历，遇到收藏时间早于 1 小时前（`now - 1h`）的记录立即停止；采用时间窗口覆盖（非 videoId 去重），保证定时器延迟时不漏数据
- **对标账号自动创建规则**:
  - 遇到未知博主（secUserId 不存在于 DB）时，调用 fetchUserProfile 后自动创建 BENCHMARK_ACCOUNT
  - 同一 secUserId **全局唯一**（不按员工隔离），`userId` 设为第一个触发该博主收藏的员工（first-creator 语义），`organizationId` 取该员工的组织
  - 已归档的对标账号（`deletedAt != null`）：同一博主被再次收藏时，**不自动重建、不自动恢复归档状态**，直接跳过
- **边界**:
  - 去重：同一博主不重复创建对标账号
  - 爬虫需先爬取收藏列表，再爬取对应博主信息
  - 收藏列表可能很长，需增量同步
  - 单账号爬虫失败不中断整体扫描，跳过该账号并记录日志
- **验收标准**:
  1. 员工在抖音收藏视频后，5 分钟内系统自动创建该博主的对标账号
  2. 对标账号展示博主基本信息
  3. 同一博主被多个员工收藏时，对标账号只创建一次，userId 为最早触发的员工
  4. 时间窗口外（收藏时间 > 1 小时前）的视频不触发新对标账号创建

### F-002-2: 对标账号管理 (P0)

- **描述**: 展示和管理所有对标账号及其视频内容，支持手动添加、归档（软删除）及查看归档列表
- **交互**: 对标账号页 → 博主卡片列表 → 点击进入 → 视频列表 → 可选择视频进行 AI 拆解
- **数据**: DouyinAccount (BENCHMARK) + DouyinVideo[]
- **手动添加**:
  - 任意员工可手动输入博主主页链接添加对标账号（复用 F-001-1 的爬虫预览 Drawer，`type=BENCHMARK_ACCOUNT`）
  - secUserId 全局唯一去重：已存在未归档账号时提示「该对标博主已存在」；已归档时提示「该对标博主已被归档，请前往归档列表查看」
- **软删除（归档）**:
  - 仅账号创建者（`douyinAccount.userId == session.userId`）可执行归档操作；后端 API 层双重校验，非创建者返回 403
  - 归档后 `DouyinAccount.deletedAt = now()`，账号和视频数据保留（不物理删除）
  - 归档后该账号不再参与定时同步；收藏同步遇到同 secUserId 时跳过（不重建、不恢复）
- **归档列表入口**:
  - 主列表页提供「查看已归档」文字链接（低调但可见，避免干扰主操作）→ `/benchmarks/archived`
  - 归档列表展示所有已归档（`deletedAt IS NOT NULL`）的对标账号及其视频（只读，数据不再更新）
  - v0.2.x 不提供硬删除和恢复归档操作
- **边界**:
  - 对标账号的视频数据也需要定时同步（同 F-001-2 机制）
  - 视频旁显示"已拆解"/"未拆解"状态标记
- **验收标准**:
  1. 对标账号列表展示关键信息
  2. 视频列表中可快速识别哪些已拆解
  3. 可从视频直接发起拆解操作
  4. 手动添加相同 secUserId 时有明确去重提示，阻止重复创建
  5. 员工只能归档自己创建（userId 匹配）的账号，他人账号无归档入口
  6. 归档后账号从主列表消失，数据保留；主列表页有「查看已归档」入口

## M-003: AI 拆解

### 概述

对对标账号的视频进行 AI 转录和文案拆解。拆解模式为"划句+批注"——在完整文案上做多层重叠高亮标注。人工修正后的拆解结果作为 AI 仿写的"框架"，同时作为经验回馈 AI。

### F-003-1: 视频转录 (P0)

- **描述**: AI 将短视频内容转录为文字文案
- **交互**: 在对标视频详情页 → 点击"AI转录"→ 调用 AI → 显示转录文案 → 可人工校对
- **数据**: Transcription (视频ID、转录文本、AI模型、状态)
- **边界**:
  - 转录可能不完全准确，需支持人工编辑
  - 无语音的视频处理（标题/字幕提取）
  - 转录失败提示
- **验收标准**:
  1. 点击转录后生成文案文本
  2. 文案文本可人工编辑修正

### F-003-2: AI 拆解 — 划句批注 (P0)

- **描述**: AI 对转录文案进行结构化拆解，以"高亮划句+侧边批注"形式展示
- **交互**:
  1. 左侧显示完整文案文本
  2. AI 自动在文本上标记多个高亮区域
  3. 同一段落/句子可能被多次高亮（不同长度、不同含义）
  4. 每个高亮对应一条批注（纯文本），显示在右侧或弹出
  5. 人工可修改：拖拽调整高亮范围 / 编辑批注文本 / 删除批注 / 新增批注
- **数据**: Analysis (转录ID、状态) + AnalysisAnnotation[] (起始位置、结束位置、批注文本、排序)
- **边界**:
  - 高亮可重叠（同一文本多个批注）
  - 人工修正后需保存两个版本：AI原版 + 人工修正版
  - 修正后的版本作为后续 AI 拆解的 few-shot 训练样本
- **验收标准**:
  1. AI 拆解后文案上出现多处高亮和对应批注
  2. 可拖拽/编辑/增删高亮和批注
  3. 修正后可保存并标记为"已审核"

### F-003-3: 拆解结果管理 (P0)

- **描述**: 管理所有拆解结果，作为仿写框架库
- **交互**: 拆解列表页 → 筛选（按账号/状态/时间）→ 查看详情 → 选择作为仿写框架
- **数据**: Analysis (视频ID、状态[AI生成/已修正/已审核]、关联转录)
- **边界**:
  - 只有"已修正"或"已审核"状态的拆解可作为仿写框架
  - 列表需展示拆解的简要摘要（前N字+批注数量）
- **验收标准**:
  1. 拆解列表可按多条件筛选
  2. 可从拆解直接跳转到仿写流程

## M-004: 碎片观点

### 概述

全公司共享的纯文本观点池。任何员工可录入碎片化观点（金句、论据、案例、数据等），作为 AI 仿写时的"血肉"填充到拆解产生的"框架"中。

### F-004-1: 观点录入 (P0)

- **描述**: 员工录入碎片化观点
- **交互**: 观点库页 → 点击添加 → 输入文本内容 → 提交
- **数据**: Fragment (文本内容、创建者ID、创建时间)
- **边界**:
  - 单条观点有字数上限（建议 500 字）
  - 支持批量录入（多条换行分割）
- **验收标准**:
  1. 可成功录入观点
  2. 观点出现在全公司共享列表中

### F-004-2: 观点浏览与选择 (P0)

- **描述**: 在仿写流程中浏览和选择碎片观点
- **交互**: 观点库列表 → 搜索/滚动浏览 → 勾选多个观点 → 确认选择 → 传入仿写流程
- **数据**: Fragment[]（被选中的观点列表）
- **边界**:
  - 全公司共享，全量可见
  - 支持文本搜索
  - 展示创建者信息
- **验收标准**:
  1. 可搜索和浏览全部观点
  2. 可多选观点用于仿写

## M-005: AI 仿写

### 概述

核心产出功能。以拆解框架为骨架、碎片观点为血肉，由 AI 生成完整短视频文案。支持多模型、多轮生成、版本管理。AI 风格训练绑定到抖音账号维度。

### F-005-1: 发起仿写 (P0)

- **描述**: 选择框架 + 观点 + 模型，发起 AI 仿写
- **交互**:
  1. 选择一个拆解结果作为框架（来自 M-003）
  2. 选择 N 个碎片观点（来自 M-004）
  3. 选择目标抖音账号（决定 AI 风格）
  4. 选择 AI 模型
  5. 点击"生成" → AI 产出完整文案
- **数据**: Rewrite (框架ID、模型ID、目标账号ID、状态) + RewriteVersion (版本号、生成文案、使用的观点IDs、使用的模型)
- **边界**:
  - 框架必须为"已修正/已审核"状态
  - 生成过程可能耗时，需 loading 状态
  - 生成失败需提示并可重试
- **验收标准**:
  1. 选完框架+观点+模型后能成功生成文案
  2. 生成的文案完整可读

### F-005-2: 版本管理与迭代 (P0)

- **描述**: 对同一个仿写任务多次生成，管理多个版本
- **交互**:
  1. 仿写详情页 → 显示所有已生成的版本列表
  2. 每个版本记录：使用模型 + 选择的观点 + 生成结果
  3. 可基于不同模型/不同观点重新生成新版本
  4. 选择某个版本标记为"最终稿"
  5. 最终稿可进行人工编辑
- **数据**: RewriteVersion[] (版本号、文案文本、模型、观点IDs、是否最终稿、人工编辑稿)
- **边界**:
  - 最终稿只能有一个（切换最终稿时旧的取消标记）
  - 人工编辑稿独立存储，不覆盖 AI 原稿
  - 版本列表按时间倒序
- **验收标准**:
  1. 可多次生成不同版本
  2. 可标记/切换最终稿
  3. 最终稿可人工编辑且保留 AI 原稿

### F-005-3: Prompt 模板管理 (P0)

- **描述**: 管理员配置 AI 各步骤的 Prompt 模板
- **交互**: 管理后台 → Prompt 模板列表 → 创建/编辑模板 → 指定适用步骤（转录/拆解/仿写）→ 指定适用模型
- **数据**: PromptTemplate (名称、步骤类型、模板内容、关联模型、状态)
- **边界**:
  - 模板中需支持变量占位符（如 {{framework}}, {{viewpoints}}, {{style_examples}}）
  - 每个步骤+模型组合至少需要一个默认模板
  - 模板变更不影响已生成的历史结果
- **验收标准**:
  1. 管理员可创建和编辑 Prompt 模板
  2. AI 各步骤使用对应模板进行调用

## M-006: 发布复盘

### 概述

员工将视频发布到抖音后（人工操作），在平台上将发布的视频与仿写稿关联。平台爬取发布后的视频数据，提供趋势图。最终将 AI 仿写稿 + 人工修正稿 + 数据表现 三者关联，作为 AI 风格训练的经验数据。

### F-006-1: 视频-文案关联 (P0)

- **描述**: 员工将"我的账号"中的已发布视频与某个仿写最终稿手动关联
- **交互**: 我的账号 → 视频详情 → "关联文案"按钮 → 选择一个仿写任务（的最终稿）→ 确认关联
- **数据**: VideoRewriteLink (视频ID、仿写ID、关联时间)
- **边界**:
  - 一个视频只能关联一个仿写稿
  - 一个仿写稿可以被多个视频关联（同稿多平台/多账号）
  - 取消关联
- **验收标准**:
  1. 视频详情中能看到关联的文案
  2. 仿写详情中能看到关联的视频和数据表现

### F-006-2: 数据趋势展示 (P0)

- **描述**: 展示已发布视频的数据变化趋势
- **交互**: 视频详情 → 数据面板 → 折线图展示播放量/点赞/评论/转发随时间变化
- **数据**: DouyinVideo 的历史数据快照 (VideoSnapshot: 视频ID、采集时间、播放量、点赞数、评论数、转发数)
- **定时器**: 独立的视频快照采集定时器，**每 10 分钟**全量扫描所有 DouyinVideo，写入 VideoSnapshot 记录
- **边界**:
  - 需要定时任务持续采集数据快照（非一次性）
  - **采集粒度：10 分钟/次**，每次写入一条 VideoSnapshot 记录，不覆盖历史
  - 时间轴支持 10 分钟粒度折线图展示（可聚合为小时/天视图）
  - 初始数据少时图表的展示处理（空状态提示或单点展示）
  - 历史数据按时间倒序存储，支持按时间范围查询
- **验收标准**:
  1. 视频数据以折线图展示（播放量/点赞数/评论数/转发数各一条曲线）
  2. 可看到数据随时间的变化趋势（10 分钟粒度）
  3. 定时器每 10 分钟自动执行一次快照采集

### F-006-3: 经验反馈闭环 (P0)

- **描述**: 将 AI 仿写稿 + 人工修正稿 + 视频数据表现 关联起来，作为 AI 风格训练的经验数据
- **交互**: 系统自动构建经验记录。员工可在仿写详情中看到"经验状态"
- **数据**: StyleExperience (账号ID、仿写ID、AI原稿、人工修正稿、视频ID、数据表现摘要、质量评分)
- **边界**:
  - 经验绑定到抖音账号维度（一个员工多个账号各自独立风格）
  - 当仿写稿关联视频且视频有数据后自动生成经验记录
  - 经验数据在后续仿写时作为 few-shot 示例注入 Prompt
  - 数据表现越好的经验权重越高
- **验收标准**:
  1. 关联文案+视频+数据后自动生成经验记录
  2. 后续仿写时能引用该账号的历史经验
  3. 经验质量与数据表现关联

## AI 系统设计要求

### 多模型支持

- 支持配置多个 AI 厂商/模型（如 OpenAI、Claude、通义千问等）
- 不同步骤可配置不同默认模型（转录用模型A，拆解用模型B，仿写用模型C）
- 员工在仿写时可选择模型

### Prompt 管理

- 管理员可通过后台配置每个步骤的 Prompt 模板
- Prompt 模板支持变量注入
- 转录 Prompt、拆解 Prompt、仿写 Prompt 分别管理

### 风格训练机制

- 基于 few-shot learning：将历史经验（优秀案例）作为 Prompt 中的示例注入
- 风格维度绑定到抖音账号（非员工）
- 随经验积累，AI 生成内容越来越符合该账号风格
- 经验筛选策略：数据表现好的经验优先被选为示例

## 爬虫 API 技术规格

> 爬虫服务为独立部署的第三方 REST API，文档地址 `{CRAWLER_API_URL}/docs`。

### 基础信息

| 属性 | 值 |
|------|----|
| **服务地址** | 环境变量 `CRAWLER_API_URL`（开发默认 `http://47.96.227.116:8011`） |
| **协议** | HTTP REST / JSON |
| **核心标识** | `sec_user_id`（抖音用户安全ID，所有后续接口的入参） |

### 接口清单

| # | 路径 | 用途 | 关键入参 | 关键出参 |
|---|------|------|----------|----------|
| 1 | `/api/douyin/web/get_sec_user_id` | URL → secUserId | `url` (抖音主页链接) | `sec_user_id` |
| 2 | `/api/douyin/web/handler_user_profile` | 获取账号资料 | `sec_user_id` | 昵称、头像、粉丝数、作品数、简介等 |
| 3 | 视频列表接口 | 获取用户视频列表 | `sec_user_id`, 分页参数 | 视频列表 (videoId, title, cover, stats…) |
| 4 | `/api/douyin/web/fetch_user_collection_videos` | 获取用户收藏视频 | `sec_user_id` | 收藏视频列表及所属博主信息 |
| 5 | `/api/douyin/web/fetch_one_video` | 获取单条视频详情 | `aweme_id` (视频ID) | 视频最新播放数据 (播放量、点赞、评论、转发) |

### 对接策略

- **字段映射**: 爬虫接口返回的 JSON 结构较复杂，采用**日志驱动**方式对接。所有爬虫响应先完整打印 JSON 日志，由开发者检查日志后逐步完善字段映射
- **文件存储**: 视频封面和视频文件 URL 从爬虫获取后需下载到本地。当前阶段使用本地文件系统存储，后续迁移 OSS
